# Экономическая эффективность: архитектура чанков TAT

Хранение на основе чанков TAT сокращает использование памяти и токенов в **25–30 раз** по сравнению с сырыми markdown/JSON/CSV логами, сохраняя 100% значимой информации благодаря структурному приоритету связей над признаками (`connection_weight = 4`).

## Реальный сценарий: агентная система с постоянной памятью

**Система:** Типичный LLM-агент с постоянной памятью сессий на протяжении нескольких месяцев работы.
**Форматы до TAT:** Markdown логи сессий, CSV трейсы оценок, JSON causal index, периодические отчёты о здоровье.

### До TAT (сырое хранение)

| Метрика | Значение |
|---------|----------|
| Сессий в месяц | ~600 |
| Данных на сессию | ~10 КБ (markdown + CSV) |
| Общий объём (4 месяца) | ~24 МБ |
| Всего токенов | ~12 млн |
| Ежемесячные затраты на LLM* | ~$24.00 |

*При анализе 100% данных через LLM по цене $2/1M токенов.

### После TAT (чанковая архитектура)

| Метрика | Значение |
|---------|----------|
| Данных на сессию (сжато) | ~0.33 КБ |
| Общий объём (4 месяца) | **<1 МБ** |
| Всего токенов | **~400 000** |
| Ежемесячные затраты на LLM* | **~$0.80** |

**Экономия: в 30 раз по памяти, в 30 раз по затратам на LLM.**

### Дополнительная экономия: карусель чанков

Карусель чанков передаёт в LLM только **тёплые чанки** (с высоким causal density):

- LLM анализирует лишь ~20% данных
- Дополнительное **снижение затрат в 5 раз**
- **Итоговая месячная стоимость: ~$0.16** для той же системы

## Архитектура, стоящая за экономией

1. **Пятислойный чанк** (Theme, Role, Emotion, Meaning, Goal) — отделяет признаки от связей
2. **Четыре формата плотности** (pkl, bin, json, txt) — только ядро (bin) необходимо
3. **Зеркальная временная метка** — прямой машинный доступ, ноль парсинга
4. **Приоритет связей** — потеря 56% признаков без потери качества

## Применимость

Любые системы с сессионными данными:
- Агентные платформы идентичности
- Логи поддержки клиентов
- Continual learning бенчмарки
- LLM-приложения с долгой памятью
